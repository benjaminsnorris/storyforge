Architecture

Key Components

Four interlocking parts that make self-healing work.

Healing Zones

Wrap risky operations in a protected block. Inside the zone, set -e is disabled and an ERR trap captures failures instead of halting the script.

API: begin_healing_zone() / end_healing_zone()

Error Handler

When a command fails, the ERR trap fires and captures the last 50 lines of stderr plus the last 30 lines of the log file for diagnostic context.

Trap: _sf_zone_error_handler

Diagnosis Engine

Sends a structured prompt to Claude (Sonnet model) with the error context, failed command, exit code, and project location. Claude diagnoses and fixes files in-place.

Function: _run_healing_attempt
Model: Sonnet via select_model

Retry Logic

After each healing attempt, the zone re-executes the original operation. Up to 3 total attempts before the zone gives up and exits with an error.

Max attempts: 3
On exhaustion: zone exits with error

Logging

Log File Structure

Every healing session is logged for debugging and auditing.

working/logs/
  • working/logs/
  • healing-20260312-143022.log — main session log with timestamps, zone names, and outcomes
  • healing-attempt-1.log — Claude's diagnosis and fix for attempt 1
  • healing-attempt-2.log — Claude's diagnosis and fix for attempt 2
  • healing-attempt-3.log — Claude's diagnosis and fix for attempt 3 (if needed)

Self-healing runs automatically — no user intervention required. When a healing zone succeeds on retry, the script continues as if the error never happened. Override the diagnosis model with STORYFORGE_MODEL if needed.
