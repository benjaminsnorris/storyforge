# Storyforge Publish Skill

You are helping an author publish their book to the Bookshelf app. This uses the Bookshelf API to push content — no local bookshelf repo or database credentials needed.

## Locating the Storyforge Plugin

The Storyforge plugin root is two levels up from this skill file's directory (this skill's directory -> `skills/` -> plugin root).

Store this resolved plugin path for use throughout the session.

## Step 1: Read Project State

Read the following files to assess readiness:

- `storyforge.yaml` — project title, author (used to derive the slug)
- `reference/chapter-map.csv` — must exist and be fresh (all active scenes mapped)

Check for optional artifacts:
- `working/dashboard.html` — manuscript visualization dashboard
- Cover image at `production/cover.*` or `manuscript/assets/cover.*`

## Step 2: Check Environment

The Bookshelf API requires these environment variables:

| Variable | Purpose |
|----------|---------|
| `BOOKSHELF_URL` | Deployed bookshelf URL (e.g. `https://bookshelf.example.com`) |
| `BOOKSHELF_EMAIL` | Admin user email for Supabase auth |
| `BOOKSHELF_PASSWORD` | Admin user password for Supabase auth |
| `BOOKSHELF_SUPABASE_URL` | Supabase project URL |
| `BOOKSHELF_SUPABASE_ANON_KEY` | Supabase anon/publishable key |

Check that these are set. If any are missing, tell the author which ones are needed and how to set them (e.g. `export BOOKSHELF_URL=https://...`). Do not proceed until all are configured.

## Step 3: Regenerate Dashboard (MANDATORY)

**Always regenerate the dashboard — even if `working/dashboard.html` already exists.** Older dashboards may have been generated by a legacy bash script with broken heredoc markers.

```bash
cd <project_dir> && ./storyforge visualize
```

This overwrites `working/dashboard.html` with the current multi-page visualization. If scoring data exists, it will be included automatically.

**Do not skip this step.** The dashboard HTML is embedded in the publish manifest and sent to the Bookshelf API.

## Step 4: Publish via API

Delegate to the `storyforge publish` command. Always offer two options:

> **Option A: Run it here**
> I'll launch the command in this conversation.
>
> **Option B: Run it yourself**
> Copy this command and run it in a separate terminal:
> ```bash
> cd <project_dir> && <plugin_path>/storyforge publish [flags]
> ```

### Flags

| Flag | Effect |
|------|--------|
| `--cover` | Include the cover image (base64-encoded) in the publish |
| `--no-dashboard` | Skip dashboard inclusion |
| `--skip-visualize` | Use existing dashboard.html instead of regenerating |
| `--annotations` | Fetch and display reader annotations after publishing |
| `--dry-run` | Generate manifest without actually publishing |

Default behavior (no flags): publish content + dashboard, no cover.

Include `--cover` only if the author explicitly asks to publish the cover.

If running here (Option A):

```bash
cd <project_dir> && <plugin_path>/storyforge publish --skip-visualize
```

Use `--skip-visualize` when running here because you already regenerated the dashboard in Step 3.

If the author wants the cover too:

```bash
cd <project_dir> && <plugin_path>/storyforge publish --skip-visualize --cover
```

Wait for the author's choice. If Option B, provide the full command and end.

## Step 5: Report Results

The publish command reports:
- Chapters and scenes published
- Word count
- Dashboard inclusion status
- Cover upload status
- Highlight preservation stats (unchanged, re-anchored, orphaned)

Relay this information to the author.

If the publish fails, show the error. Common issues:
- **400**: Invalid manifest — check for missing fields or stale chapter map
- **401**: Auth failed — check environment variables
- **403**: User is not an admin — verify the Supabase user has admin role
- **500**: Server error — the response includes a `phase` field showing where it failed

## Step 6: Annotations (Optional)

If the author wants to see reader annotations (highlights and notes), run:

```bash
cd <project_dir> && <plugin_path>/storyforge publish --annotations --dry-run
```

Or call the annotations endpoint directly via the bookshelf module:

```python
from storyforge.bookshelf import authenticate, check_env, get_annotations
env = check_env()
token = authenticate(env['BOOKSHELF_SUPABASE_URL'], env['BOOKSHELF_SUPABASE_ANON_KEY'],
                     env['BOOKSHELF_EMAIL'], env['BOOKSHELF_PASSWORD'])
data = get_annotations(env['BOOKSHELF_URL'], token, '<slug>')
```

Annotations can inform revision work — reader highlights show which passages resonated or confused.
